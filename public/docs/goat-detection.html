Built for livestock operations managing chute throughput, this system replaces manual counting with an always-on, tamper-proof AI pipeline that produces audit-ready records.

Manual goat counting is error-prone, time-consuming, and impossible to audit after the fact. This system automatically counts every animal passing through your chute — in real time, on live camera feeds — and writes an immutable CSV record you can share with buyers, vets, and regulators.

Core Capabilities

Feature Deep-Dive

Every feature is engineered for production farm environments — not controlled lab conditions.

🧠
Dual-Line Counting (Recommended)

The most accurate mode. Two parallel virtual lines (A + B) work together — requiring a goat to cross both in the same direction within a time window before registering a count. Eliminates almost all false positives from animals near the line.

📹
RTSP / Live Stream Support

Connect to any RTSP, RTMP, HTTP, or webcam source. The multi-threaded pipeline captures, paces, infers, and displays simultaneously with no dropped frames from inference lag.

🎯
YOLO + ByteTrack / BoT-SORT

Uses Ultralytics YOLO for frame-level detection and either ByteTrack or BoT-SORT (with Re-ID) for persistent tracking across frames. Custom-trained model ships with the package (best.pt).

🔁
Thread-Resilient Pipeline

Capture, pacing, inference, and display run on independent threads with dedicated queues. The monitor thread auto-restarts crashed inference or pacing workers once — keeping the system live.

📡
WebRTC Browser Streaming

Watch the annotated live feed from any browser at http://host:8081. Supports up to 2 simultaneous clients, configurable resolution and FPS downscaling for bandwidth control.

🗂️
Slot Control API

FastAPI REST endpoint to start/stop named counting sessions (slots) per vendor. Each slot gets its own event CSV and a summary JSON comparing declared vs. counted animals.

🎨
Rich Visualization HUD

Live overlay shows bounding boxes or centroids, track IDs, directional arrows, count HUD, motion trails, and count-event animations — all configurable via YAML.

⚡
GPU-Accelerated Inference

FP16 half-precision on CUDA, Conv+BN fusion, configurable image size. Runs on CUDA:0 by default. Automatically disables FP16 on CPU to prevent errors.

🛡️
Anti-Flicker & Motion Gating

Two-stage verification: geometry (line-crossing confirmation) then motion intent (displacement, axis, consistency). Rejects animals that are stationary, reversing, or twitching near the line.

🔄 Auto-Recovery
The monitor loop polls thread health every 1 second. If the inference or pacing thread dies, it attempts one clean restart. If the capture thread dies, it logs the crash and attempts reconnection with exponential backoff.

Intelligence

Counting Algorithms

Three modes with increasing robustness. For chute monitoring, Dual-Line is always recommended.

Mode 1
Single-Line Counting

A single virtual line in the ROI. Counts when a track's confirmed side flips across the line. Uses side-history deque for anti-flicker and per-ID cooldown frames to prevent double-counts.

Mode 2★ Recommended
Dual-Line Counting

Two parallel lines (A + B). Verify mode requires A then B in the same direction within a frame window. Recover mode counts at A, with B catching any A misses. Then motion gating confirms the crossing is real.

Mode 3
Zone Counting

Rectangular zone — tracks entry and exit. Useful for pens or holding areas. Configurable policy for animals born inside the zone (count_entry or ignore). Supports near-border back-projection.

Dual-Line Logic — How It Actually Works

LINE A
LINE B
🐐
→ Direction: UP

Live simulation — goat crossing Line A then Line B in the same direction triggers a count event

Phase 1: Geometry (Eyes)

For every track on every frame, the system computes its cross-product sign relative to Line A and Line B using the perpendicular distance formula. Positions within line_margin_px (default 8px) are discarded as ambiguous. A deque of recent sign values is kept per track per line.

A "side flip" is only confirmed when the deque meets strict/loose majority criteria: at least min_nonzero_abs non-zero samples must agree on a direction with min_nonzero_ratio consistency. This prevents noise from triggering false flips.

Phase 2: Motion Gating (Brain)

After geometry confirms a crossing event, the MotionIntentAnalyzer independently examines the track's position history and checks:

📏 Displacement Check

Total displacement (Euclidean) must exceed motion_min_displacement_px. Rejects animals that crossed by swaying in place.

📐 Axis Alignment

The dominant movement axis (X or Y) must exceed motion_axis_min_displacement_px. Ensures motion is perpendicular to the line, not parallel to it.

🔄 Consistency Ratio

Frame-to-frame directional steps along the dominant axis must be consistent (same sign) at motion_dir_consistency_ratio (e.g., 70%). Rejects animals reversing mid-cross.

🎯 Direction Agreement

The motion-derived direction must agree with the geometry-derived direction. If they contradict, the count is rejected.

⚠️ Decision Outcomes
The analyzer returns one of three decisions: accept (count is committed), reject (motion contradicts geometry — discarded permanently), or defer (insufficient motion history yet — will re-evaluate next frame). All decisions are written to goat_decisions.csv for auditing.

Auto Profiles (Slipping vs Standing View)

The system ships with two pre-tuned parameter sets for the two most common camera angles on chute setups. Set counting.dual.profile in config and all motion/geometry thresholds are automatically applied:

Parameter standing_view slipping_view Effect
hyst_frames 4 5 Frames needed to confirm a side flip
motion_min_frames 6 8 Min motion history before analyzing
motion_min_displacement_px 14.0 16.0 Min pixel travel required
motion_dir_consistency_ratio 0.60 0.60 Required directional step consistency
id_lock_frames 70 80 Cooldown frames after a count event

REST Interface

Slot Control API

A FastAPI server (port 8090 by default) enables external systems — farm management software, POS systems, mobile apps — to control counting sessions per vendor or batch.

📚 Interactive Docs
When the API is running, visit http://127.0.0.1:8090/docs for interactive Swagger documentation auto-generated by FastAPI.
POST /api/slot/start Start a counting session

Starts a new slot. Idempotent if the same slot_id is already active (returns 200). Rejects with 409 if a different slot is active. 

{
  "slot_id": "vendor_ahmed_batch_01",
  "vendor_id": "V-001",
  "vendor_name": "Ahmed Livestock",
  "declared_count": 50,
  "started_by": "operator_1"
}
POST /api/slot/stop Stop active session & generate summary

Stops the active slot, finalizes the CSV, and generates a slot_summary.json comparing declared_count vs counted_count. Status will be OK, MISMATCH, or ABORTED. 

{
  "slot_id": "vendor_ahmed_batch_01",
  "stopped_by": "operator_1",
  "reason": "batch complete"
}
GET /api/slot/active Current slot state

Returns real-time slot state — live count, direction breakdown, elapsed time. Safe to poll at any interval. 

{
  "active": true,
  "slot": {
    "slot_id": "vendor_ahmed_batch_01",
    "slot_count": 23,
    "direction_breakdown": { "up": 20, "down": 3 },
    "declared_count": 50,
    "status": "ACTIVE"
  }
}
GET /health Health check

Returns {"status": "healthy", "service": "slot-api"}. Use for load-balancer health checks or monitoring dashboards.

Data Artifacts

Output Files

Every run generates a complete audit package under outputs/runs/<run_id>/. All outputs are timestamped and organized per run.

📋 Events CSV Primary Audit
events/goat_cross_events.csv

One row per counted crossing. Columns: timestamp_s, src_frame_idx, proc_frame_idx, track_id, direction, cx, cy. Append-only. Written atomically — duplicate track IDs are blocked at the CSV layer.

📈 Timeseries CSV Analytics
timeseries/goat_counts_timeseries.csv

Per-second cumulative counts. Columns: time_s, up_count, down_count. One row per second of video processed. Useful for throughput graphs and time-of-day analysis.

🧪 Decisions CSV Debug
decisions/goat_decisions.csv

Motion gating decisions. Columns: ts_s, tid, geometry_direction, decision, reason, confidence, dx, dy, dominant_axis. Written for every geometry event — even rejected ones. Invaluable for tuning.

⚡ Metrics CSV Performance
metrics/goat_metrics.csv

Thread-level performance metrics: FPS per stage, queue fill rates, inference latency. Used to diagnose pipeline bottlenecks and validate pacing configuration.

🎬 Annotated Video Visual Proof
video/output.mp4

Full-frame MP4 with all overlays: bounding boxes, track IDs, motion trails, line overlays, count HUD, and flash animations on each count event. Visual verification of every count.

📦 Slot Summary JSON Vendor Record
slots/SLOT_<id>__<ts>/slot_summary.json

Per-slot JSON generated on stop. Contains: declared_count, counted_count, difference, slot_status (OK / MISMATCH / ABORTED), direction_breakdown, slot_start_time, slot_end_time. Immutable once written.

Common Questions

Frequently Asked Questions

Answers to the most common technical and operational questions.

Why am I getting wrong direction counts (up vs down reversed)? +
Direction is determined by the side of the line a track is on before and after crossing. If up and down are reversed, your camera orientation doesn't match the configured profile. Try switching counting.dual.profile between standing_view and slipping_view. If neither works, manually adjust the line coordinates in counting.line_roi — swapping the line endpoints (ax,ay ↔ bx,by) inverts the direction convention.
Counts are lower than expected — animals being missed. What do I do? +
Check the decisions/goat_decisions.csv file first. If you see many "reject" entries with reasons like "motion too small" or "axis displacement too small," lower the motion thresholds: motion_min_displacement_px (try 10–12), motion_axis_min_displacement_px (try 8), and reduce motion_dir_consistency_ratio (try 0.55). If the YOLO detector itself is missing animals, lower inference.conf (try 0.15–0.20) and increase inference.imgsz if animals are small in frame.
The system is counting too many false positives. How do I reduce them? +
Increase the motion hardening parameters: raise motion_min_displacement_px (try 18–20), increase motion_dir_consistency_ratio (try 0.75–0.80), and increase hyst_frames (try 6–8). Also check that min_age (default 60 frames) is large enough — this prevents very new tracks from counting. If using single-line mode, switch to dual-line mode which has significantly lower false-positive rates.
The RTSP stream keeps disconnecting or stuttering. What should I try? +
Enable runtime.autoskip: true and reduce runtime.max_lag_s to 0.5 so the system drops old frames quickly when it falls behind. Check that your RTSP URL is correct and the camera is accessible from the machine running the software. If using WiFi, wired Ethernet significantly improves stability. The system's capture thread automatically attempts reconnection on drop — check the log for "CAPTURE — reconnecting" messages.
Can I process multiple camera feeds simultaneously? +
The current architecture supports one source per process. To process multiple cameras, run multiple instances of python main.py with different config.yaml files pointing to different RTSP sources, different Slot API ports (8090, 8091, etc.), and different output directories. Each instance is independent and fully self-contained.
How does the Slot API handle network failures or missed stop calls? +
If the process is terminated while a slot is active (power cut, Ctrl+C, crash), the shutdown handler automatically calls abort_active_slot_if_any(). This finalizes the CSV and generates a slot summary with status ABORTED. The slot directory remains on disk with all events recorded up to the point of failure. No data is lost. Duplicate start/stop calls are idempotent — the API returns 200 and logs a warning rather than throwing an error.
What is the difference between Verify mode and Recover mode in dual-line counting? +
Verify mode: A count is only committed when a track crosses Line A and then Line B in the same direction within the frame window. More conservative — you'll never count an animal that didn't definitively cross both lines. Better for slow-moving animals with clean crossing trajectories.

Recover mode: The system counts on Line A crossing immediately. Line B then acts as a recovery mechanism — if an animal somehow missed Line A (tracking gap, occlusion), it can still be counted when it crosses Line B. Better for fast-moving animals or environments with frequent occlusion. This is the default and recommended mode.
How do I set the ROI and line positions for my specific camera? +
All coordinates are normalized (0.0 to 1.0 ratios). For the ROI: xr and yr define the top-left corner, wr and hr define width and height — all as fractions of the full frame. For example, yr: 0.15, hr: 0.85 crops the top 15% of the frame.

For the counting line: "ax,ay,bx,by" where each value is relative to the ROI (not the full frame). A vertical line at 18% across the ROI spanning full height would be "0.1875, 0.0, 0.1875, 1.0". Take a screenshot from your camera and measure in a tool like Paint or GIMP to find the right ratios.

🔍 Log Analysis
All events are logged to logs/log.jsonl in structured JSON format. Each line includes timestamp, level, tag, and message. Filter by tag (e.g., grep "DUAL-MOTION-REJECT" log.jsonl) to diagnose specific pipeline stages. The decisions CSV is especially useful for tuning — it records every geometry event with its motion gating outcome.
